About pd-vm and RustScript

Overview

pd-vm is a stack-based virtual machine plus compiler toolchain used as a backend for multiple source syntaxes, including our primary focus: RustScript (.rss).

Features include:

  • Executes compiled compact bytecode rather than interpreting source.
  • Offers consistent runtime semantics for both synchronous and asynchronous execution.
  • Includes rich debugging and profiling tools: interactive debugger, recording and replay, and JIT trace insights.

How To Use

Fuel Metering

pd-vm provides Wasmtime-style fuel controls on both Vm and Store<T>.

Fuel metering is disabled by default. If there is not enough fuel, execution returns VmStatus::Yielded before the next instruction runs.

Compiler Internals for RustScript

The compiler uses compile_source() for RustScript or compile_source_file() for flavor selection.

Supported Syntax and Features

  • Statements: use, fn, pub fn, let, assignment, indexed/member assignment, if/else, while, C-style for, break, continue.
  • Expressions: literals (int, float, bool, string, null), arithmetic, logical, comparison, function calls, closures (|...| expr), if-expression form (if cond => { ... } else => { ... }), match expressions.
  • Match patterns: int/string/null literals, wildcard _, and type constructors Some(TypeName) / Option::Some(TypeName).
  • Collections: array literal [], brace literals for arrays/maps, obj.member, obj[key], optional chaining (?. and ?.[key]), slice syntax ([a:b], [:b], [a:]), map key literals including null.
  • Host/runtime calls: namespaces io::..., re::..., json::..., jit::..., imported via use vm::{...};.

Ownership and Liveness Behavior

  • let bindings are immutable by default; use let mut for mutable bindings.
  • Reassignment, indexed/member mutation, and &mut borrows require a mutable binding.
  • Non-copyable field reads (e.g., p.a) and literal index element moves (e.g., let x = arr[0]) are move-by-default and clear the moved field/element on the owning container.
  • Non-copyable local-to-local rebinds (e.g., let b = a) move the source local by default.
  • RustScript function calls apply consumed-parameter inference; reusing a consumed caller local after the call is rejected.
  • Use .copy() to explicitly clone/detach a value.
  • &expr and &mut expr are accepted borrow forms. In the current subset, they act as non-consuming access forms but participate in move/liveness checks.
  • Collection aliases are tracked through assignment, borrow binding, and passthrough calls; mutating a collection while an alias exists is rejected unless detached (e.g., via .copy()).
  • Locals must be definitely available on every control-flow path before use.
  • Closures and fn declarations capture outer locals at definition/declaration time. Capture mode follows expression semantics (x can move, x.copy() copies, and &x / &mut x keep borrow-style capture behavior).
  • Inline fn and closure calls clear per-call frame slots after producing the call result. Closure capture slots participate in liveness null-clears once the owning closure/value is dead.

Current RustScript Compiler Limitations

  • crate::... module paths are not supported in source loading; use relative module paths.
  • Callable locals can be passed and called, but are not runtime Values and cannot be stored in arrays/maps or returned.
  • Recursive RustScript function declarations are not supported by current inlining-based lowering.
  • Function declarations implicitly capture outer locals (snapshot at declaration time).
  • In move-semantics mode, implicit captures follow expression semantics.
  • match patterns are limited to int/string/null literals, _, and type constructors (Some(TypeName)).
  • break and continue are only valid inside loops.
  • Host import namespace support allows vm, io::, re::, json::, and jit::.

Directives include .data, .code, .label NAME, and .local NAME [INDEX].

Builtins and Bridged `call` Opcode

  • Builtin calls: fixed reserved indices emitted by parser lowering (e.g. len, assert, io::*).
  • Runtime host imports: per-program remapped indices emitted as call <slot>, <argc>.
  • Inlined RustScript function bodies: calls to targets with FunctionImpl are inlined without emitted call opcodes.

JIT Internals

The trace-based JIT path detects hot bytecode loops, records straight-line trace semantics, and emits native machine code via Cranelift, executing trace semantics without bytecode re-decoding.
